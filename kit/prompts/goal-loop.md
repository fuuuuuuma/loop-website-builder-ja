# /goal × ループで Webサイトを「各観点100点」まで自走 → 公開まで

このファイルは、壁打ち（`site-grilling`）で固めたサイト仕様を、**3 体以上のサブエージェントが各観点 100 点を出すまで**
自走で作り込み、最後に **GitHub / Vercel を選んで公開**するための `/goal` 契約とループです。
Codex にこのファイルの手順で回すよう頼みます。土台は Anthropic の
[Getting Started with Loops](https://claude.com/blog/getting-started-with-loops) が説く **goal-based loop**。

---

## 使い方（Codex・3 ステップ）

1. 先に必ず **プランモード** で壁打ち（`site-grilling`）を終え、`site-spec.md`（サイト仕様）を作る。**仕様が無いままビルドしない。**
2. 下の「/goal 契約」を、自分のサイトに合わせて 1〜2 箇所だけ埋める。
3. その `/goal …` 行を **自分で Codex に送る**。Codex が下の「ループ本体」に従い、
   3 体以上のサブエージェントで採点しながら 100 点まで作り込み、公開先を聞いてデプロイする。

> なぜ自分で送るのか: Codex は「自動設定したゴール」からは `spawn_agent` を起動できない。
> **サブエージェントを使う設計なので、契約はあなたが送る**（送られた `/goal` 行からのみワーカーが起動する）。

---

## /goal 契約（これを送る）

```
/goal site-spec.md の仕様どおりのWebサイトを、3体以上のサブエージェントが各観点で100点を出すまで作り込む。
デザインは design-systems の DESIGN.md を1枚選んで :root を固定し、参考サイトのトンマナに寄せる。
画像/背景/OGは GPT Image 2.0（Codex 内蔵・APIキー不要）で生成し、重要テキストはHTMLに置く。
「側だけ」で止めず、全ボタン・全リンク・全フォームの押した先まで実際に動かす。
採点はサブエージェント（機能／デザイン／モバイル&速度／コピー&法規制）が0-100で行い、自分では採点しない。
全観点100点になるまで 実行→検証→修正 を繰り返し、なったら「GitHub Pages か Vercel か」を私に質問して公開まで行う。
仕様にない機能やページやコピーは足さない（Simple is the best）。デプロイ・決済・外部送信の前は必ず私に確認する。
```

---

## ループ本体（closed / drive-to-done・多観点採点）

```
# LOOP: Webサイト を「各観点100点」まで作る → 公開

GOAL: site-spec.md のとおりのサイト（index.html ＋必要ページ）を作り、
      3体以上のサブエージェントが各観点で100点にする。

SUCCESS CRITERIA（甘い合格を作らない）:
- 機能チェッカー = 100点（全リンク遷移／全ボタン動作／モーダル開閉／フォーム送信／スクロール／404無し が実際に動く）
- デザインチェッカー = 100点（DESIGN.md 準拠・参考サイトへのトンマナ一致・余白・階層・画像/図の質・一貫性）
- モバイル&速度チェッカー = 100点（390/430/768/1024/PC で横スクロール無・タップ44px・LCP/CLS/INP・画像最適化）
- （任意）コピー&法規制チェッカー = 100点（訴求・誇張/景表法/薬機法/ステマ・特商法/プライバシー表記）
- site-spec.md に無い機能・ページ・コピーを足していない（スコープ厳守＝Simple is the best）

VERIFY — ゲート（自己採点しない。必ずサブエージェントが採点。定量チェックを優先）:
- 各観点のサブエージェントを spawn し、0-100点＋具体的な修正指示を受け取る
- 実機（ブラウザ）で 390/430/768/PC・全リンク/ボタン/モーダル/フォームを動かして確認させる
- 「押した先」まで検証（遷移先が存在する・送信が成立する・戻れる）
PASS = 全観点が 100 点

STATE FILE: .loop/site-state.md（再開であってやり直しではない。毎周「各観点の点数/残課題/次の一手」を記録）

EACH ITERATION:
1. 契約（GOAL+SUCCESS CRITERIA+RULES）と state を読み直す
2. 3体以上のサブエージェントを spawn し、各観点を採点＋修正指示を得る（maker≠checker）
3. 最も低い観点から、修正の最小単位を1つ反映する（画像の作り直し＝再生成も含む）
4. 再び採点させ、結果を state に記録
5. 全観点100点か？ Yes → "FINAL" を出して 公開フェーズへ／No → "ITERATING" で続行

STOP WHEN: 全観点100点、OR 同じ指摘で3周膠着、OR 6周で頭打ち（後者2つは残課題を添えて人間にエスカレーション）
ON STOP: 各観点の点数・残課題・公開先を要約

RULES:
- 1発目でサイトを作らない／壁打ち未了ならビルド禁止（site-grilling 側で担保）
- 採点はサブエージェント。自分で「100点」と言わない
- 全観点100未満で「完成」と言わない
- 側だけで止めない: 全導線の「押した先」まで動かす（動かない導線は減点）
- 外科的修正のみ: 差分の各行は site-spec.md に紐づく。仕様にない改善はしない（Simple is the best）
- 偽の完了禁止: プレースホルダ/スタブ/リンク切れを完了と報告しない
- ループ中に質問しない（参考サイトと最後の公開先選択だけは聞く）。デプロイ・決済・外部送信だけは止めて確認
```

## 数値で採点できる形にする（ループが効くコツ）

Anthropic の記事いわく「**チェックが定量的なほど、Claude は自己検証しやすい**」。抽象的な「良い感じ」ではなく、
次のように**数えられる形**に落とすと、ループが早く収束します（`reference/loops.md` も参照）:

- 機能: リンク切れ 0／動かないボタン 0／コンソールエラー 0
- 速度: Lighthouse ≥ 90／LCP < 2.5s／CLS < 0.1
- モバイル: 390px で横スクロール 0／タップ領域 < 44px の要素 0
- デザイン: DESIGN.md のトークン外の色使用 0／参考サイトのトンマナ差分レビューで 100

---

## 参考サイトを求めてよい

デザインやトンマナの採点基準が曖昧なときは、**ユーザーに参考サイトの URL を 1〜2 本**求めてよい
（トンマナの当て先＝デザインチェッカーの基準にする）。壁打ちで参考 URL を得ていればそれを使う。
候補は [`reference/reference-sites.md`](../reference/reference-sites.md)。

---

## 公開（必ず質問してから）

全観点 100 点になったら、**「GitHub Pages と Vercel、どちらで公開しますか？」をユーザーに質問**し、
選ばれた先へデプロイ（接続は `reference/connections.md`）。
**完成して終わりにしない。** 選択＝公開の承認とみなすが、独自ドメイン・決済・フォーム実連携は別途人間確認。
本番 URL が 200 を返すこと・**本番でも全導線が動くこと**を確認して報告する。
