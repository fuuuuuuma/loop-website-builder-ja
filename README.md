# ループエンジニアリングで「完璧なWebサイト」を1発生成する方法 徹底解説

> AIに「サイト作って」と一度頼むだけでは、なぜ“それっぽいガワ”で止まってしまうのか。
> その答えが **ループ（loop）**。作る人と採点する人を分けて、合格するまで回す——この考え方で、
> ボタンの押した先まで動く「完璧なWebサイト」を仕上げる方法を、はじめての人にも分かるようにまとめました。
> （AIエージェントチャンネル・解説用リサーチノート／視聴者は初心者を含む前提）

📊 [スライド資料を見る](./slides.html) ｜ 📄 [1枚まとめ資料を見る](./onepager.html) ｜ 🎁 [プレゼントを受け取る](./present.html) ｜ 📦 [キット一式](./kit/)

## TL;DR（3行）

- **1発で完璧なサイトは出ない。** AIは“動くと思う”ものを返すだけ。だから **「合格の定義」と「採点役」を用意して、達成するまで繰り返す＝ループ** で仕上げる ✅
- Webサイト作りに要るのは3つ。**①参考サイトでトンマナを決める ②使うコンポーネントを知る ③壁打ち→`/goal`→サブエージェント採点で100点まで回す** ✅
- 大事な2つの合言葉。**「側だけ作らない（ボタンの押した先まで動かす）」** と **「Simple is the best（目的1つ・要素を盛らない）」** 🔶

## 1. ループエンジニアリングとは何か

### まず一言で

**ループ**とは、AIエージェントが「**止まる条件に達するまで、作業のサイクルを自分で繰り返すこと**」です ✅。
人間が毎回「これ直して」「次これやって」と舵を取るのではなく、**AIが自分で作って・自分で確かめて・直す**を回します。

この考え方は、Anthropic（Claudeの開発元）の公式ブログ
**「Getting Started with Loops」**（2026年6月30日公開・Delba de Oliveira / Michael Segner）で整理されています ✅。
なお、この一連の設計は「ループエンジニアリング」とも呼ばれます。公式ブログはループを **4つのパターン** に分けており（次項）、この資料では**そのうちの「ゴール型」を使う前提**で説明します。

### なぜ「ループ」なのか — 1回の会話の限界

AIに「いいねボタンを作って」と頼むと、記事の言葉を借りればこうなります ✅:

> 「AIはコードを読み、編集し、テストを走らせ、**"動くと思う"ものを返す**」

問題はこの **“思う”** の部分。合っているかどうかを**人間が毎回チェックして、直しを指示する**必要があります。
サイト全体ともなると、このチェックと指示の往復が積み重なって、時間も気力も削られます。
**ループは、この「毎回人間が確認する」ボトルネックを、AI自身の検証に置き換えます。**

### ループの種類（公式ブログの分類）

公式ブログは、ループを**きっかけ・止まり方・使う道具**で4種類に分けています ✅:

| 種類 | きっかけ | 止まる条件 | 使う道具 | 向いている仕事 |
|---|---|---|---|---|
| **ターン型** | あなたの一言 | AIが「できた」と判断 | スキル（検証手順を教える） | 短い・単発の作業 |
| **ゴール型（`/goal`）** | あなたが手で送る | ゴール達成 or 上限ターン | 成功条件＋採点役 | **ゴールが明確な作業（サイト制作）** |
| **時間型（`/loop`・`/schedule`）** | 時間・間隔 | 完了 or 停止 | ローカル/クラウドの繰り返し | 定期・監視 |
| **プロアクティブ型** | イベント・予定 | 目標達成まで自動 | 上記の合わせ技＋ワークフロー | 定型の運用フロー |

Webサイト制作で使うのは **ゴール型（`/goal`）** です。「各観点100点」という合格条件を決めて、
達成するか上限まで、作って→採点して→直す、を繰り返します。

### 効くループの3原則

公式ブログが挙げる、うまくいくループのコツはこの3つです ✅:

1. **自分で検証できる形にする。** 「良い感じ」ではなく**数えられるチェック**に。
   記事いわく *「チェックが定量的なほど、Claudeは自己検証しやすい」*。
   → 例：リンク切れ0／Lighthouse 90以上／スマホ幅で横スクロール0。
2. **作る人と採点する人を分ける（maker ≠ checker）。** 自分の作品を自分で甘く採点しない。
   **別のサブエージェント（AIの分身）に採点させ、100点になるまで直す。**
3. **シンプルから始める。** 記事いわく *「すべての作業に複雑なループは要らない。まず一番簡単な解から」*。
   小さな仕事に多段ループは過剰です。

> ここが全体の土台です。以降の「壁打ち」「`/goal`」「サブエージェント採点」は、
> この3原則をWebサイト作りに当てはめただけ、と考えると迷いません。

## 2. なぜ Webサイト作成に「ループ」が必要なのか

Webサイトは、**「見た目」と「機能」の両方が揃って初めて完成**します。ところが、一発生成が苦手なのは主にこの機能面です。

- **見た目（ガワ）はそれっぽく出る。** ヒーロー、料金表、フッター……AIは形をすぐ作れます。
- **でも“中身”が抜けやすい。** ボタンを押しても何も起きない、リンクが `#`（どこにも飛ばない）のまま、
  スマホで横にはみ出す、フォームが送信できない——**「動いていない」箇所**が残りがちです。

この「動いていない箇所」は、**人間が一つずつ触って確認しないと気づけません**。まさに第1章の“思う”問題。
だから、**「全リンクが飛ぶ・全ボタンが動く・全フォームが送れる」を合格条件にして、そこに達するまで回す**必要があります。

さらにWebは**採点を数値化しやすい**のが好都合です ✅:

- リンク切れ **0** 件／コンソールエラー **0** 件
- Lighthouse スコア **90以上**／表示速度（LCP）2.5秒未満
- スマホ幅（390px）で横スクロール **0**／タップ領域 44px 以上

こうした**数えられる基準**をゲートに置けるので、ループが早く収束しやすくなります。
「なんとなく良い」で止めず、**「合格の線」を数字で引ける**——これがWebサイト×ループの相性の良さです。

### ループの一周を、具体例で

イメージしやすいように、料金セクションを作るときの「一周」を追ってみます 🔶:

1. **作る（generator）**：AIが料金表を3カードで組み、CTAボタンを置く。
2. **採点する（checker＝別のサブエージェント）**：機能担当が実機で確認 → 「CTAボタンを押しても何も起きない（リンク未設定）。**機能60点**」。モバイル担当 → 「390pxで3カードがはみ出す。**モバイル70点**」。
3. **直す**：最も低い機能から。CTAに実際の申込先を配線し、カードをスマホで縦積みに。
4. **もう一度採点**：機能100点・モバイル100点・デザイン95点（余白の指摘）。
5. **まだ100点でない観点を直す** → 全観点100点になったら次のセクションへ。

ポイントは、**採点したのが「作った本人」ではない**こと。自分で「たぶんOK」と流さないから、
「押しても動かないCTA」のような**側だけ問題**が見つかりやすくなります。これが maker ≠ checker の効き目です。

## 3. AIでWebサイトを作るのに必要な3つ

「いきなり作らせる」から「準備してから回す」へ。必要なのは次の3つです。

### ① 参考サイトをチェックする（トンマナの当て先を決める）

作る前に、**近い雰囲気のサイトを1〜2本決めます**。これが後で「デザインの採点基準」になります。
（基準がないと、AIも人も「良い/悪い」を判断できず、ループが空回りします。）

探すのに便利なギャラリー（一例）🔶:

- **SANKOU!**（https://sankoudesign.com/ ）— 国内サイトを、種類・業種・色・フォント・動き・レイアウトで細かく分類。日本語サイトの当て先探しに便利 ✅
- **Awwwards / Godly / Land-book / One Page Love / siteInspire** — 海外の最先端・厳選ギャラリー
- **Mobbin** — 実アプリのUIパターン集

> 使い方は「良いものをスクショして URL を控える」だけ。**レイアウトのコピーではなく、トンマナ（雰囲気）の参考**にします。
> 一覧は配布キットの [`kit/reference/reference-sites.md`](./kit/reference/reference-sites.md) に入れています。

### ② コンポーネントの種類を知る（部品の名前を覚える）

サイトは「**ページ全体のセクション**」と「**その中の小さなUI部品**」の2階層でできています。
**部品名を知っておくと、AIに「ここは○○を使って」と一言で正確に頼めます** ✅。

- **セクション**：ナビバー／ヒーロー／実績バー／特徴グリッド／料金表／お客様の声／FAQ／CTA／フッター
- **UI部品**：ボタン、タブ、アコーディオン、モーダル（ダイアログ）、カルーセル、トースト、チップ、フォーム……

部品の分類は、Googleの **Material 3（m3.material.io/components）** が辞書として便利です ✅
（マテリアル調の見た目を真似る必要はなく、**名前と役割を知る**目的で使います）。
まとめは [`kit/reference/components.md`](./kit/reference/components.md) に。

### ③ 壁打ち → `/goal` → サブエージェント採点で回す

ここが本題。手順はこうです:

1. **AIとの壁打ちから始める。** いきなり作らせず、**一問一答**で「誰に・何のために・何をしてほしいか（ゴール1つ）」と
   **各ボタンの押した先**を固めます（＝仕様書 `site-spec.md`）。この壁打ちの型は、
   Matt Pocock の **GrillMe**（MIT）という有名な手法をWebサイト向けに作り替えたものです。
2. **`/goal` で検証と修正を繰り返す。** 「各観点100点になるまで」という成功条件を渡して、AIに自走させます。
3. **サブエージェントに評価させる（合格するまで動かす）。** 自分で採点させず、
   **機能／デザイン／モバイル&速度／コピー&法規制**の担当を分けて0〜100点で採点。
   **全部が100点になるまで**、作って→採点して→直す、を回します（maker ≠ checker）。

> この3つを回すためのプロンプト・スキル一式が、この記事のプレゼント（配布キット）です（第8章）。

## 4. 「側だけ」作ってもダメ — 押した先まで作り込む

一番やりがちな失敗が、**見た目のガワ（＝側）だけで満足してしまう**ことです。
きれいなヒーローや料金表ができると完成に見えますが、**押した先が動かなければサイトとして未完成**です。

チェックすべき「押した先」🔶:

- **リンク**：ヘッダー・CTA・フッターの全リンクが、実在する行き先へ飛ぶ（`href="#"` のダミーを残さない）
- **ボタン**：押すと実際に反応する（モーダルが開く／タブが切り替わる／送信される／該当位置へスクロール）
- **フォーム**：送信先が配線されている（メール・外部フォーム等）。未定なら「準備中」モーダルで**閉じられる**ように
- **仕掛け**：ハンバーガーメニュー・アコーディオン・スムーススクロールが、触って反応する
- **戻れる**：モーダルや別ページから、ちゃんと元に戻れる

だからループの合格条件には、必ず **「全リンク・全ボタン・全フォームの押した先まで動く」** を入れます。
サブエージェントには「**実機で一つずつ押して確かめる**」ところまでやらせます。
ここまでやって初めて「完璧なWebサイト」と言えます。

## 5. Simple is the best — 盛らない勇気

もう一つの合言葉が **「Simple is the best」**。

- **目的は1つ、CTA（してほしい行動）は1つ**に絞る。あれもこれも載せると、どれも刺さりません。
- 「この要素は**唯一のゴールに効くか？**」を毎回問い、効かないものは**削る**。
- アニメーション・色・セクションを**盛りすぎない**。迷ったら減らす。余白も情報です。

公式ブログも同じ趣旨で *「まず一番簡単な解から始める」「小さな仕事に複雑なループは要らない」* と言っています ✅。
シンプルな作りは、速く・崩れにくく・直しやすく・ループも早く収束します。**引き算がそのまま品質になる**、と覚えてください。
逆に、要素が多いサイトは「動かない箇所」も「崩れる箇所」も増え、採点が下がりやすく、ループが長引きます。
**「載せる理由を1つずつ説明できるか」**を基準にすると、自然と削れて、結果的に速く完成します。

## 6. 使い方（初心者基準・コピペでOK）

むずかしい設定は要りません。**あなたは“監督”**。やってほしいことを言葉で伝え、採点はAIの分身に任せる——それだけです。
（ランタイムは Codex デスクトップアプリを想定。ChatGPTアカウントで動き、画像生成も内蔵・APIキー不要 🔶）

### こんな悩み → こう頼むだけ

| こんな悩み | こう頼む（コピペ可） |
|---|---|
| いきなり作られて“ガワ”だけになる | 「今すぐ作らないで。まず**一問一答で壁打ち**して、仕様を固めてから見せて」 |
| ボタンが動かない・リンクが飛ばない | 「**全リンク・全ボタン・全フォームを実機で押して**、動かない所とリンク切れを一覧で出して」 |
| 見た目がバラバラ | 「配色は**DESIGN.mdを1枚選んで固定**、参考サイト（URL）のトンマナに寄せて」 |
| 自分では良し悪しが分からない | 「**サブエージェントに機能/デザイン/モバイルを0-100で採点**させて、100点まで直して」 |
| 盛りすぎて重い | 「目的に効かない要素は削って。**Simple is the best** で」 |

### そのままコピペで試せるプロンプト

**まず壁打ちから（1発で作らせない）:**

```
これからWebサイトを作りたい。ただし今すぐ作らないで。
まず一度に1問ずつ・各問に推奨案をつけて、私に壁打ちして。
「目的（1つ）／見せる相手（1人）／ゴール(CTA1つ)／必要なセクション／各ボタンの押した先／参考サイト」
を埋めて、仕様としてまとめてから見せて。
```

**ループで100点まで（サブエージェント採点）:**

```
/goal 仕様どおりのWebサイトを、3体以上のサブエージェントが各観点で100点を出すまで作り込んで。
観点=機能／デザイン／モバイル&速度／コピー。採点はサブエージェントが0-100で行い、自分では採点しない。
「側だけ」で止めず、全ボタン・全リンク・全フォームの押した先まで動かして（href="#"を残さない）。
要素は足しすぎない（Simple is the best）。100点になったら公開先(GitHub/Vercel)を私に聞いて。
```

まずはこの2つだけで、「壁打ち → ループ → 採点」の流れが体験できます。

### よくあるつまずき Q&A

**Q. 結局、普通に「作って」と頼むのと何が違うの？**
A. 「作って」だけだと、AIは“動くと思う”ものを1回返して終わりです。ループは**合格条件（例：全ボタンが動く／横スクロール0）を決めて、達成するまで自分で直す**ところが違います。人間のチェック回数が減ります ✅。

**Q. サブエージェント（AIの分身）って難しそう…**
A. やることは「**採点する係を分ける**」だけです。「機能・デザイン・モバイルを別々に0-100点で採点して、100点まで直して」と頼めば、あとはAIが役割を分けて回します。あなたは点数と残りの課題を見るだけ。

**Q. 何回くらい回るの？終わらないのが不安。**
A. 止め時を先に決めておきます。「**全観点100点**」で終了、または「同じ指摘で3周」「6周で頭打ち」なら一度止めて相談、が目安です 🔶。暴走もムダ回しも防げます。

**Q. デザインの正解が分からない。**
A. だから**参考サイトを1〜2本**先に決めます（第3章）。それが採点の基準になります。配色は `DESIGN.md` を1枚選んで固定すれば、見た目が勝手にブレません。

**Q. 公開まで全部AI任せで大丈夫？**
A. 公開・決済・フォーム連携など**取り消せない操作の前は必ず人間が確認**します。ここは自動化しないのが安全です。

## 7. もう一歩進んだ使い方

慣れてきた人向けに、少しだけ踏み込んだ話です（本筋は上の6章までで十分です）。

- **成功条件を数値で書く。** 「Lighthouse 90以上／リンク切れ0／390pxで横スクロール0」のように、
  **数えられる形**にするほどループが早く収束します（公式ブログの原則）✅。
- **状態ファイルで“再開”にする。** ループの各周で「各観点の点数・残課題・次の一手」を1ファイルに記録すると、
  やり直しではなく**続きから**進められます。
- **止め時を決める。** 「全観点100点」か「同じ指摘で3周」か「6周で頭打ち」で止め、後者は人間に相談する——
  暴走とムダ回しを防ぎます。
- **画像とテキストの役割分担。** 背景・装飾・写真・OG画像は画像生成（GPT Image 2.0 等）に、
  **見出し・価格・日付・CTAはHTMLテキスト**に。後から直せて、SEOにも崩れにも強くなります 🔶。
- **公開は必ず人間が確認。** デプロイ・決済・フォーム連携など**取り消せない操作**の前は、必ず一度止めて確認します。
